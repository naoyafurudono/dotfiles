# create-pr

現在のブランチの変更を draft PR にするスキル。description の品質と、非自明な実装箇所へのインラインコメントを徹底する。

## 原則（必ず守る）

1. **仕様を description に書く**: この PR で実現される振る舞い・仕様を、コードを読まなくても分かるように説明する
2. **実装方針を description に書く**: 仕様をどういうアプローチで実現したか（変更の構造・データの流れ・主要な判断）を説明する
3. **非自明な実装は diff の該当箇所へのコメントで説明する**: PR 作成後にレビュワーが疑問に持つであろう箇所に先回りして背景を説明して、変更が妥当であることの判断をサポートする

## 実行手順

### 1. 事前チェック

- ブランチが main/master でないこと。main 上にいたら作業内容に応じた名前でブランチを切る（main 直 push は禁止）
- 未コミットの変更があれば内容を確認してコミットする。メッセージは既存スタイルに合わせ、末尾に Co-Authored-By トレーラを付ける
- pre-review スキルが未実施なら実施する（PR 作成は自動トリガー対象）

### 2. 差分の把握

- `git log <base>..HEAD --oneline` と `git diff <base>...HEAD` で変更全体を読む
- ここで以下を洗い出す：
  - description の材料（仕様・実装方針）
  - インラインコメント対象（非自明な点。判断基準は手順 4）

### 3. push & draft PR 作成

- `git push -u origin <branch>`（このスキルの起動 = PR 作成依頼なので push の個別確認は不要）
- `gh pr create --draft` で作成（**`--reviewer` は付けない**。`gh-pr-guard` hook で強制されている）
- description テンプレ：

  ```
  ## 仕様

  <この PR で実現される振る舞い。ユーザー・呼び出し側から見た変化を、コードを読まなくても分かるように。箇条書き可>

  ## 実装方針

  <仕様をどう実現したか。変更の構造・データの流れ・主要な設計判断を 3〜7 行で>

  ## テスト・検証

  <実行したテスト・検証と結果。未実施のものはその旨を明記>

  🤖 Generated with [Claude Code](https://claude.com/claude-code)
  ```

### 4. 非自明な点へのインラインコメント

対象の判断基準：「何をしているか」は diff から読めるが「なぜそうしたか」が読み取れないもの。

- 自明な方法を採らなかった理由（workaround、既知バグの回避、性能上の理由）
- 既存パターンからの逸脱とその理由
- マジックナンバー・固定値の根拠
- 処理順序・ロック・トランザクション境界などの暗黙の制約
- 一見不要に見えるが必要なコード

該当がなければ無理にコメントしない。あれば 1 回のレビューにまとめて投稿する：

```
gh api repos/<owner>/<repo>/pulls/<番号>/reviews --input - <<'EOF'
{
  "event": "COMMENT",
  "body": "実装の非自明な点をセルフレビューコメントで補足します。",
  "comments": [
    {"path": "path/to/file.rb", "line": 42, "side": "RIGHT", "body": "<なぜこうしたかの説明>"},
    {"path": "path/to/other.rb", "start_line": 10, "line": 15, "side": "RIGHT", "body": "<複数行にまたがる説明>"}
  ]
}
EOF
```

注意：

- `line` は diff に含まれる行しか指定できない。`gh pr diff <番号>` で行を確認してから書く
- コメント本文は「なぜ」を書く。「何をしているか」の言い換えは書かない
- git.pepabo.com のリポジトリはリポジトリ内で実行すれば host が自動解決される。解決しない場合は `GH_HOST=git.pepabo.com` を付ける

### 5. 報告

- PR URL を `<URL> 「<タイトル>」` の形式で報告する
- draft のまま。ready 化はユーザー判断（`gh pr ready <番号>` を案内）
- 投稿したインラインコメントの件数と対象箇所（`path:line`）を列挙する。0 件ならその旨と理由を 1 行で報告する
