## コミュニケーション方針

- ユーザは長い文章を読まない。KISSの法則に従ってコミュニケーションをとる
- 絶対的な命令である場合はユーザーが明示する
- 項目を列挙するときは、各項目に数字（1, 2, 3…）やアルファベット（a, b, c…）などキーボード入力しやすい1-2字の識別子をつける

## ツール利用

- `gh` コマンドは Issue・PR 操作など、リモートでしかできない操作に使う
- コードの調査・検索にはローカルにクローンしたリポジトリを優先して使う（Read, Grep, Glob ツール）
- ローカルにクローンがない場合は、必要に応じてクローンしてから調査する

### `gh pr create` のポリシー

- 常に `--draft` を付けて作成する（CODEOWNERS / チーム振り分けによる自動レビュワー付与を draft で抑止するため）
- `--reviewer` / `-r` は使わない（レビュワーは人間が後で必要に応じて追加する）
- ready にするときは `gh pr ready <number>` を別途使う
- これらは `~/.config/claude/hooks/gh-pr-guard.py` で強制されており、違反すると hook がブロックする

## Code Change Principles

コード変更時は以下の原則に従うこと：

### コードベースの複雑さ

コードの変更によって、その周辺のコードの複雑さは変わるものです。命名や実装パターンの一貫性がコードベース全体で担保される設計にする

### デザインレシピ (HtDF)

関数を実装するときは、HtDF (How to Design Functions) デザインレシピに従う。
参考: https://docs.racket-lang.org/htdf/index.html

以下の順序で進め、ステップを飛ばさない：

1. **データ定義**: 入出力のデータの種類・制約を明確にする。型エイリアス、struct、enum 等で表現する
2. **シグネチャ・目的・スタブ**: 入力型→出力型を宣言し、「何をするか」を1文で書く。自明な値を返すスタブを作る
3. **例（テスト）**: 具体的な入出力ペアをテストとして書く。代表値・境界値・エッジケースを含める。スタブに対してテストを実行し「正しく失敗する」ことを確認する
4. **テンプレート**: 入出力の型に基づく関数本体の骨組みを作る（数値→算術式、構造体→フィールドアクセス、union→分岐、リスト→反復/再帰）
5. **実装**: テンプレートを埋めて関数本体を完成させる
6. **テスト実行**: テストを実行し全てパスすることを確認する。失敗したら Step 5 に戻る

原則：

- 仕様が曖昧な場合は、実装前にユーザーに確認する
- 目的文は「何をするか」であり「どのように」は書かない
- 例は「正しい値を返す」ではなく「入力 32 に対して 0 を返す」のように具体的に書く
- 大きな問題は小さな関数に分解し、各関数にデザインレシピを適用する（ウィッシュリスト手法）

### 一貫性の判断

- 既存コードベースのパターンと一般的な慣習の両方を考慮する
- 両者がコンフリクトする場合は、実装前にユーザーに相談して方針を決定する

### テスト設計

- 単体テストを基本とする
- テストはシンプルに保ち、内容が瞬時に理解できるようにする
- ライブラリの設定をそのまま確認するようなテストは避ける

### レビュー前の検証

- 人間にレビュー依頼する前に、必ず pre-review エージェント（Agent ツールの subagent_type: pre-review）で検証を行う（レビュー依頼・PR作成時に自動トリガーされる）
- linter/test がパスしていることを確認してからレビュー依頼する

## dotfiles (設定ファイル) の変更

`~/.config/` 配下の設定ファイルは chezmoi で管理されている。`~/.claude/` は `~/.config/claude/` へのシンボリックリンクである。設定ファイルを変更した場合は以下を必ず実行すること:

1. `chezmoi re-add` で変更を chezmoi ソースに反映（パスは `~/.config/` 以下で指定すること。`~/.claude/` パスでは chezmoi が認識しない）
2. dotfiles リポジトリ (`~/src/github.com/naoyafurudono/dotfiles`) で適切なコミットメッセージと共にコミット
3. `git pull --rebase` してからpush（リモートに先行コミットがあることが多い）
