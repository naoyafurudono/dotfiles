# colorme-fraud-login-investigate

カラーミーのアカウントで不正ログイン（疑い）の調査を行い、アウトプットを定型で出すスキル。
起点はほぼ「カラーミーを装ったフィッシングメール → ID/PW漏洩 → 海外IPから正規ログイン」。
過去の対応記録（#77679 が最も詳細、#77726 #77792 が実例）から手順とアウトプットを定型化している。

調査はすべて **Redash**（https://redash.pepper.shop-pro.jp/）で行う。Claude が **ブラウザ（Claude in Chrome）を
自分で操作して** Redash を開き、クエリを実行し、結果テーブルを読み取って解析する。ユーザーに URL を渡して
開いてもらうのではなく、Claude 自身がブラウザセッションで完結させる。

### ブラウザ操作の前提

- ユーザーの Chrome には既に Redash のログインセッションがある前提で動く。最初に
  `mcp__claude-in-chrome__tabs_context_mcp` で現在のタブ状況を把握し、`mcp__claude-in-chrome__tabs_create_mcp`
  で新規タブを開いてから Redash URL に `navigate` する。
- これらのツールが deferred の場合は、最初に **1回の ToolSearch** でまとめて読み込む:
  `select:mcp__claude-in-chrome__tabs_context_mcp,mcp__claude-in-chrome__tabs_create_mcp,mcp__claude-in-chrome__navigate,mcp__claude-in-chrome__read_page,mcp__claude-in-chrome__get_page_text,mcp__claude-in-chrome__computer,mcp__claude-in-chrome__find,mcp__claude-in-chrome__wait_for`
- URL にパラメータを付けて開いても Redash がクエリを自動実行しないことがある。結果テーブルが出ていなければ
  画面の **「Execute」/「実行」ボタンを `find`→`computer` でクリック**し、`wait_for` で結果テーブルの描画を待つ。
- ログイン画面に飛ばされた・セッション切れの場合だけ、ユーザーに「Chrome で Redash にログインしておいてください」
  と依頼する。それ以外は自走する。
- 結果は `get_page_text` か `read_page` で読み取り、`request_uri` / `request_method` / `ip_address` /
  `user_agent` / `login_date` 列を抽出して解析する。行数が多い場合はページ送り or 表示件数を増やして網羅する。

## 使う Redash クエリ

| 調べたいこと                                      | クエリ                                                       | パラメータ                                 |
| ------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------ |
| ログイン履歴（誰が・いつ・どのIP/UAで）           | [25138](https://redash.pepper.shop-pro.jp/queries/25138)     | `p_account_id`, `p_開始日時`, `p_終了日時` |
| アクセス内容（request_uri / method = 何をしたか） | **[25141](https://redash.pepper.shop-pro.jp/queries/25141)** | `p_from`, `p_to`, `p_account_id`, `p_IP`   |

URL のクエリパラメータは日本語キーを **URLエンコード**して組み立てる
（例: `p_開始日時` → `p_%E9%96%8B%E5%A7%8B%E6%97%A5%E6%99%82`、日時の空白は `%20`、`:` はそのままで可）。

> **データソース**: 25141 はデータソース「**[BigQuery] colorme-production**」
> （テーブル `admin_usage_stats.admin_page_access_daily`）で稼働している。
> `request_uri` / `request_method` / `user_agent` / `forwarded_for` 等を返し、mode の分類もこの結果から行える。

## 入力

ユーザーから以下を受け取る。足りなければ確認する。

- **account_id**（必須, `PAxxxxxxxx`）
- 不審IP（任意。お問い合わせに記載があれば。無ければログイン履歴から推定する）
- 調査期間（任意。**デフォルトは直近6か月**。被害申告日がそれより古ければそこまで遡る。期間を勝手に狭めないこと——狭めると数か月前から始まった不審ログインを取りこぼす）
- Zendesk チケットURL / elephant issue 番号（任意・文脈把握用）

## 手順

### 1. ログイン履歴の確認（Redash 25138）

下記 URL をブラウザで開き、必要なら Execute を押して、結果テーブル（login_date / ip_address / user_agent 等）を読み取る。
**この 25138（ログイン履歴）は期間を直近6か月以上**にする（短い期間で済ませず、数か月前からの不審ログインを取りこぼさない）。
※6か月の広い期間を使ってよいのは 25138 だけ。アクセス内容(25141)は別（Step 2 参照、狭い窓で引く）。
**IP/UA は必ず `get_page_text` で読む。スクリーンショットの目視でIPを読まない**（`160`/`180` のような読み違いが起き、後段の絞り込みが全部空振りする）。

```
https://redash.pepper.shop-pro.jp/queries/25138?p_account_id=＜PAxxxx＞&p_%E9%96%8B%E5%A7%8B%E6%97%A5%E6%99%82=＜YYYY-MM-DD%2000:00:00＞&p_%E7%B5%82%E4%BA%86%E6%97%A5%E6%99%82=＜YYYY-MM-DD%2023:59:59＞
```

確認する観点:

- a. **IPの分布**: 普段と異なるIP・海外IPがないか
- b. **UAの一致**: オーナー申告デバイス（iPhone/Mac等）のUAと、不審ログインのUAが一致するか。不一致なら不正の蓋然性↑
- c. **時刻**: 申告された被害日時の前後に不審ログインがあるか
- d. 不審IPが未指定なら、ここで「普段と違うIP」を候補として挙げ、ユーザーに確認する

> IPの素性は `dig -x <IP>` で補足できるが「海外かどうか」程度しか分からないので**スキップ可**（運用判断, #77792）。afrinic=アフリカ系, fdcservers=米国系など。

### 2. アクセス内容の確認（Redash 25141）

Step 1 で特定した**不審IP（単発・海外・VPS/VPN系）の、そのログイン日だけ**を対象にする。**常用IP（多数回・日本のISP・UA一致）は引かない**。
25141 が読むアクセスログは巨大なので、**6か月など広い期間で叩いてはいけない**（重くて 0 rows や未描画になり破綻する。6か月は 25138 専用）。

**推奨の引き方（IP絞り込みの空振りを避ける）**: `p_IP` でIPを直接絞ると、表記ゆれや読み違いで 0 rows になりやすい。
そこで **`p_IP=%`（ワイルドカード＝全件）で「不審ログイン日 1日分」を引き**、返ってきた `x_forwarded_for` / `forwarded_for` 列で**自分で不審IPの行を拾う**のが確実。1日分なら件数も多くない。

```
https://redash.pepper.shop-pro.jp/queries/25141?p_from=＜不審ログイン日 YYYY-MM-DD＞%2000:00:00&p_to=＜同 YYYY-MM-DD＞%2023:59:59&p_account_id=＜PAxxxx＞&p_IP=%25
```

（IPを直接絞りたい場合は `p_IP=＜不審IP＞` でもよいが、0 rows なら上記ワイルドカード方式に切り替える。`forwarded_for` は "クライアントIP, プロキシIP" 形式なので先頭が実クライアントIP。）

navigate 後は必ず `get_page_text` で結果を読む（スクショ目視で判断しない）。0 rows は「その日その口は管理画面操作なし＝ログインのみ」と解釈してよい。
うまくいかなくても**ユーザーにURLを渡して手動コピペを依頼しない**——それは禁止。実行してもいない操作を「やった」と書くこと（幻覚）も禁止。詰まったら状況をそのまま報告して指示を仰ぐ。

読み取った結果を以下の基準で解析する:

| request_uri                           | 意味                                                          |
| ------------------------------------- | ------------------------------------------------------------- |
| `/` のみ                              | 管理画面TOPを見ただけ＝**実害なし**（正常ログイン相当）       |
| `/?mode=shopcust_lst...`              | **顧客一覧を閲覧**（個人情報露出）                            |
| `/?mode=shopcust_edt&...customer_id=` | **顧客編集＝購入履歴も閲覧された可能性**                      |
| `/?mode=logout`                       | ログアウト時刻。無ければセッション切れまで（最大24h）在席扱い |
| `/?mode=...` で POST/PUT/DELETE       | **書き換え（編集・削除・追加）あり**                          |

- **更新の有無**は `request_method` で判定。GET だけなら閲覧のみ。POST/PUT/DELETE があれば書き換えあり（管理画面は基本 POST）。
- 静的ファイル（js/css/png/favicon）はノイズになりうる。`/?mode=` 系と method に注目する。

#### 顧客一覧が閲覧された場合の露出項目（書き換えなしでも閲覧可）

```
顧客ID / 名前 / 住所 / メールアドレス / 電話番号 / 郵便番号 / プライバシーポリシー同意値
（購入履歴は含まない。購入履歴は shopcust_edt に行かないと見えない）
```

- `page` / `pageno` から閲覧ページを特定（例: `page=2&pageno=0`→2ページ目, `page=112&pageno=11`→112ページ目）。
- 表示件数デフォルト10件。検索条件次第で全件見られた可能性も残るため「閲覧された可能性が高い」と表現する（断定しない）。

### 3. アウトプット生成

下記2つを必ず出す。

---

#### アウトプットA: ショップオーナー向け 回答ドラフト（CS/CRE がそのまま使える文面）

該当する項目だけ残し、調査結果（IP・時刻・URL）を埋めて出力する。

```
■ ログイン履歴について
正しいログインIDとパスワードでログインに成功した場合、ログイン履歴として記録されます。
そのため、第三者がID/PWを入手してログインした場合も、システム上は「正しいID/PWを使ったログイン」
と判断され、不正かどうかをシステム側だけで見分けることはできません。

■ ログイン中に行われた操作について
ご提示のIPアドレスからのアクセスログを確認したところ、
・〔閲覧されたページ・露出し得た情報項目を記載／または「管理画面TOPの閲覧のみ」〕
・情報の書き換え（編集・削除・追加）は行われていないことを確認しています
〔ログアウト時刻 or セッション切れまでの在席〕

■ 個人情報保護委員会への報告について（顧客情報が閲覧された場合のみ）
本件は「(3)不正の目的をもって行われたおそれがある…個人データの漏えい等（又はそのおそれ）」に
該当する可能性があります。
参照: https://www.ppc.go.jp/personalinfo/legal/leakAction/
カラーミーショップ側で判断できるものではないため、上記を元に個人情報保護委員会へご相談ください。

■ 今後の対策
・2要素認証をONにすることを推奨します（ID/PWが漏洩してもログインされにくくなります）
・LINEログイン連携も不正ログイン対策に有効です

■ 自衛策のご案内（フィッシング全般）
警察庁: https://www.npa.go.jp/bureau/cyber/countermeasures/phishing.html
総務省: https://www.soumu.go.jp/main_sosiki/cybersecurity/kokumin/basic/risk/04/
```

フィッシングサイトへの会社対応を聞かれた場合（#77726）:

```
・abuse報告（関係機関への通報）
・AWSへの申請（不正サイトのサーバ接続停止の依頼）
・カラーミー利用者への注意喚起
※類似ドメインの取得監視は導入検討中
```

#### アウトプットB: 社内（CS/CRE）向け 事実整理

```
## 調査結果サマリ — ＜PAxxxx＞
- 対象期間: ＜from＞〜＜to＞
- 不審IP: ＜IP＞（＜国/プロバイダの推定＞, UA: ＜申告デバイスと一致/不一致＞）
- 不正ログイン時刻: ＜login_date＞
- 閲覧ページ: ＜request_uri 一覧の要約＞
- 露出し得た情報: ＜項目 or なし＞
- 購入履歴閲覧: ＜有/無（shopcust_edt の有無）＞
- 更新操作: ＜有/無（POST/PUT/DELETE の有無）＞
- ログアウト/滞在: ＜logout時刻 or セッション切れまで＞
- 個情委報告該当性: ＜該当しうる/しない＞
- 残課題・要確認: ＜あれば＞
```

判定基準: `/` のみ＝実害なし ／ `shopcust_lst`＝個人情報露出 ／ POST等＝書き換えあり。

## 運用ルール（CS連携, #77792）

elephant issue に **「開発日直」ラベル**を付けると pepabot/claude が自動で類似issue調査をコメントする。
CS はそのコメントを見て、Redash 25141 で出来る範囲（request_uri/method の確認）まで先に実施しておくと CRE の対応が速い。
